Part 3: System Design

Chapter 5

Context Engineering is the discipline of choosing what information Claude sees at any given moment. It is not about writing longer prompts — it is about assembling the right context window.

📌 Key Concept: Context Assembly = gathering the rules, files, memos, and tool descriptions needed for the current task into the actual input. It is a selection process, not a dumping process. More context is not better context.

What fills the context window on every turn (even a one-line prompt):

  • Your current question
  • Recent conversation history
  • Files Claude has read
  • CLAUDE.md and rule files
  • Tool execution results
  • Claude's own recent output

Context Rot occurs when the context grows so long that early signals weaken. Long conversations gradually lose coherence — not because the model forgets, but because important early instructions get diluted by later noise.

⚠️ Common Mistake: Loading the entire repository into context. Even when fixing a single bug, some users dump every file. This is slower, more expensive, and produces lower quality results. Show Claude only the function and the 20 lines of error log it needs.

Three isolations to design simultaneously:

  1. Context isolation — Don't mix investigation, planning, implementation, and review in one long session. Use subagents or separate sessions.
  2. File state isolation — Use worktrees or separate output folders for conflict-prone work.
  3. Approval isolation — Separate frequently-allowed actions from always-review actions via permissions and hooks.

Harness Engineering goes beyond prompt tuning. It encompasses the entire workspace: permissions, folder structure, test policies, session resume rules, handoff protocols, worktree isolation, and remote control setup.

📌 Key Concept: A harness designs three things at once: (1) what Claude knows (context), (2) what Claude can do (tools and permissions), (3) what Claude must verify (tests and exit criteria). Missing any one of these creates a different class of failure.

The start routine matters more than the start prompt:

  1. Check resident contract — read CLAUDE.md, relevant rules, current branch
  2. Classify the task — investigation, implementation, or review?
  3. Write the plan — scope, off-limits areas, verification commands
  4. Confirm exit criteria — check tasks/verify.md
  5. Only then begin actual work
💡 Pro Tip: Subagents and worktrees solve the same problem from different angles. Subagents isolate context (delegate research or review to a temporary worker). Worktrees isolate file state (physically separate working directories). Use subagents for brief investigations; use worktrees for parallel experiments.

One of the most effective subagent uses is not making code faster, but simulating different user roles: a skeptical senior engineer, a new maintainer, a security reviewer, an ops person. Different perspectives catch trust-breaking issues like missing READMEs, side effects in status commands, and observability gaps.

Verification is what separates demos from production. A good verification plan answers: "What must pass before this task is done?"

The tasks/verify.md template:

# Verification Plan
## Must Pass
- [ ] Unit tests: `pnpm test -- signup`
- [ ] Lint: `pnpm lint`
- [ ] Type check: `pnpm typecheck`
## Must Review (human)
- [ ] Manual browser check on mobile
- [ ] CTA text approved by stakeholder
## Must Not Break
- [ ] Existing auth flow unchanged
- [ ] No new dependencies added
📌 Key Concept: Verification skills (checking that signup flows work end-to-end, that checkout states are correct, that CLI works in TTY environments) often have higher ROI than generation skills. A "well-written draft" is cheap — a "verified passing result" is expensive and valuable.
⚠️ Common Mistake: Treating test passage as the goal rather than maintainable implementation. Models can hardcode values or add special cases to make tests green. Your verification prompt should explicitly forbid shortcuts and require reporting when tests seem wrong.

Tokens are both a billing unit and a signal for how wide your task has spread. Even a one-line prompt sends much more: the question, conversation history, loaded files, CLAUDE.md, rules, tool outputs, and recent model output all enter the context window.

Token TypeWhat It Includes
Input tokensPrompts, files, conversation history, rule files, tool outputs
Output tokensExplanations, code, summaries, tool call results
📌 Key Concept: "Saving tokens" is not about being stingy — it is about designing so Claude reads only what it needs. The Prompt Cache provides reuse benefits when similar backgrounds are sent repeatedly. Structure your context so the stable prefix (CLAUDE.md, rules) stays consistent and cacheable.

Cost is managed by structure, not by counting:

  • Reduce tool output verbosity (don't dump entire logs)
  • Keep sessions focused — don't mix unrelated tasks
  • Use /compact at strategic points, not as a last resort
  • Separate investigation from implementation into different sessions
  • Track cost with /cost to build intuition
💡 Pro Tip: The Tool Result Budget concept means controlling how much context tool outputs consume. A test suite that dumps 500 lines of output eats your context fast. Configure tools to return summaries, not raw logs.

Resume Hygiene means keeping only what the next session needs and discarding accumulated noise. Long sessions are not inherently better — they accumulate context rot.

Session Memory template:

# Current State
- What I'm doing right now

# Task Specification
- What the user actually requested

# Files and Functions
- Important files and why they matter

# Workflow
- Frequently used commands and order

# Errors & Corrections
- Failed attempts — do NOT repeat these

# Key Results
- Confirmed results and answers

# Worklog
- Recent steps (keep very short)
📌 Key Concept: Long sessions survive when information converges into five categories: current state, files, workflow, errors, and results. Everything else is noise. The Errors & Corrections section is not a shame log — it is a re-failure prevention device.
💡 Pro Tip: Use the status line (/statusline) as a dashboard. It shows working directory, Git branch, and context usage. When context usage climbs, save state to files and start a fresh session — don't push through until quality degrades.
