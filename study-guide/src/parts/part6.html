Part 6: Community Patterns & Tools

Chapters 9–10

Community practitioners have developed recurring patterns for managing Claude's memory and context over time. These are not official features — they are operational habits that emerge from heavy daily use.

📌 Key Concept: Session Memory is not about saving entire conversations. It is about maintaining structured state files (handoff.md, decision-log.md, worklog.md) that the next session reads on startup. The conversation is disposable; the files are the real memory.

Common memory patterns:

  • File-based memory: CLAUDE.md + rule files + handoff files form a persistent memory layer that survives session boundaries
  • Structured recall: Starting each session by reading tasks/handoff.md and tasks/plan.md before any new work
  • Progressive disclosure: Loading background at the top, task-specific details only when activated (like skill frontmatter triggering full skill content)
💡 Pro Tip: The most effective memory strategy is the simplest: at the end of every session, update handoff.md with current state, next steps, and unresolved risks. This single habit prevents 80% of "where was I?" confusion.

The plan.md pattern is one of the most widely adopted community practices. Instead of keeping the plan in conversation, write it to a file that both Claude and humans can reference.

Why plans belong in files, not conversations:

  • Files survive session boundaries; conversations don't
  • Files can be reviewed by humans independently
  • Files can be version-controlled with Git
  • Files keep the conversation focused on execution, not re-planning
📌 Key Concept: The plan file is not bureaucracy — it is a rework-prevention device. Teams that plan before editing consistently report fewer rollbacks, shorter review cycles, and less context pollution in long sessions.

Effective plan.md structure:

# Plan
## Goal
One sentence describing the desired end state.

## Scope
- Files to modify: [list]
- Off-limits: [list]

## Steps
1. ...
2. ...

## Verification
- Tests to pass: [list]
- Manual checks: [list]

## Open Questions
- [Any unresolved items before starting]

Advanced users run multiple Claude sessions in parallel, each with a different role. This is not about speed — it is about context isolation.

Common parallel patterns:

  • Plan + Implement: One session plans, another implements based on the written plan
  • Implement + Review: One session writes code, another reviews in a separate worktree
  • Main + Subagent research: Main session stays focused while a subagent explores documentation or gathers data
  • Multi-worktree experiments: Test different approaches in isolated working directories, compare results
⚠️ Common Mistake: Parallelism without role clarity creates chaos, not speed. Before launching parallel sessions, define: who does what, which files each session can touch, and how results merge back. Unstructured parallelism is worse than sequential work.
💡 Pro Tip: The "OpenClaw-style" operation that many community members aspire to means: always-on sessions, multi-channel access (terminal, phone, web), persistent state across sessions, scheduled tasks, and remote control. In the Claude ecosystem, you achieve this by combining Claude Code + CLAUDE.md + hooks + subagents + MCP + Remote Control + Channels + Scheduled Tasks.

Community patterns include using Claude to improve its own environment — creating skills, refining rules, and evaluating past outputs.

Self-improvement loop:

  1. Run a task with current rules and skills
  2. Evaluate the output (manually or with a review subagent)
  3. Update rules, skill descriptions, or templates based on failures
  4. Re-run and compare

SEO and content automation patterns:

  • Keyword research → content structure → draft → SEO review skill → publication
  • Multi-channel adaptation: one research session feeds blog, social, newsletter, and email templates
  • Content calendar automation with scheduled tasks
📌 Key Concept: The meta-skill skill-creator can generate initial skill drafts from descriptions. However, it produces first drafts, not production-ready skills. Failure cases, edge conditions, and verification loops still require human review and iteration.

With growing ecosystems of skills, plugins, and MCP servers, selecting the right tools requires discipline.

Evaluation checklist before installing any external tool:

  1. When was it last maintained?
  2. Does it have real README and examples?
  3. What security-sensitive permissions does it need?
  4. Is the context cost justified by the utility?
  5. Can you disable it quickly if something goes wrong?
