Part 2: Practical Playbook

Chapters 3–4

The most common confusion after installation is: "Where do I start?" The answer depends on your task type.

CoworkClaude Code
Document-first tasks: meeting notes, research briefs, folder reading, draft generationDevelopment-first tasks: bug fixes, tests, refactoring, plan documents
Don't expect it to be a universal backend systemDon't start by delegating an entire new feature at once
📌 Key Concept: This chapter is not about how to use Claude Code — it is about how to operate it. The core habits are: plan-before-edit, resume hygiene, review split, worktree separation, allowlists, and structured handoffs.

Session management follows a decision tree:

  • Same goal, continuing? → claude --continue or --resume
  • Need a different direction? → --fork-session
  • File conflicts or role conflicts? → --worktree (separate working directory)
💡 Pro Tip: People who use Claude Code well over time are not people who write better prompts — they are people who make work less interruptible.

Eight habits separate productive Claude Code users from frustrated ones:

  1. Plan before edit — Fix scope, verification, and no-go zones before touching code. For big tasks, write plan.md first.
  2. Keep resident context thin — CLAUDE.md is a work contract, not an encyclopedia. Move long-term state to handoff.md, decision-log.md.
  3. Allowlist first, bypass later — Start with narrow allow and deny rules. Only widen permissions after understanding them.
  4. Manual /compact at the right time — Compress after saving important state to files, not after the session has already degraded.
  5. Use resume deliberately — --continue for same direction, --fork-session when direction splits.
  6. Separate plan / implement / review — Don't cram all roles into one session. Split reduces context pollution and approval fatigue.
  7. Use worktrees — For conflicting work, experiments, or parallel reviews, separate the working directory.
  8. Manage cost by structure, not token counting — Reduce cache misses, tool output length, session length, and test log verbosity.
⚠️ Common Mistake: Jumping straight to implementation without a plan. In big tasks, this feels fast but leads to rework. The plan is not a bureaucratic step — it is a device that reduces rework.

A well-organized repository separates responsibilities across files. Here is the harness blueprint:

repo/
├── CLAUDE.md                    # Project contract (always loaded)
├── .claude/
│   ├── settings.json            # Allow/ask/deny boundaries
│   ├── rules/
│   │   ├── architecture.md      # Structure rules
│   │   ├── testing.md           # Test standards
│   │   └── frontend.md          # UI conventions
│   └── hooks.json               # Auto-enforcement rules
└── tasks/
    ├── plan.md                  # What to do next
    ├── verify.md                # Exit criteria
    ├── decision-log.md          # Why this path was chosen
    └── handoff.md               # Current state for next session
📌 Key Concept: Each file prevents a different type of failure. CLAUDE.md prevents context drift. settings.json prevents unsafe actions. rules/* prevents domain-specific mistakes. hooks.json turns "remember to do X" into "always do X." tasks/* prevents session amnesia.

Settings scope priority: Managed > Command-line > Local > Project > User. Organization-locked rules cannot be overridden by individual settings — this is why "my setting doesn't work" confusion often happens.

Team vs. local separation: Team-shared standards go in CLAUDE.md and .claude/settings.json. Personal experiments go in CLAUDE.local.md or .claude/settings.local.json.

💡 Pro Tip: A minimal hook example that blocks dangerous Bash commands: 
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": ".claude/hooks/block-rm.sh"
      }]
    }]
  }
}

Effective prompts are not about cleverness — they follow patterns. Here are the key prompt styles:

Context-first prompt (recommended start routine):

Read CLAUDE.md, relevant rules, and tasks/verify.md first.
Do NOT implement yet. Summarize:
1. Task type
2. Modification scope
3. Off-limits areas
4. Verification commands
Then proceed only after approval.

Shortcut-prevention prompt (for coding tasks):

Goal: maintainable implementation, not just passing tests.
Rules:
1. No hardcoding or special-casing to pass tests.
2. If tests conflict with requirements, report first.
3. If using external code, explain why before importing.
4. Separate direct implementation from external references.
5. After changes, report what passed and what's unverified.
📌 Key Concept: The three engineering layers: (1) Prompt Engineering — crafting better instructions, (2) Context Engineering — choosing what to show Claude, (3) Harness Engineering — designing the entire work environment. Most people stop at layer 1. The real gains come from layers 2 and 3.

Compact Instructions template — controls what survives after /compact:

## Compact Instructions
When compacting, always preserve:
- Last modified files and why
- Passed tests, failed tests, untested items
- Rejected alternatives and reasons
- The ONE thing next turn should do immediately
- Risk points requiring human review
💡 Pro Tip: Good compression instructions don't make summaries pretty — they make the next session able to continue immediately.

Six slash commands form your session management toolkit:

📌 Key Concept: Weekly evaluation should track: which CLAUDE.md / rules / hook versions were active, which skills or templates were used, what changed from last week, and whether pass rates and human correction amounts improved. Change rules and hooks — not just prompts — to find your optimal configuration.

Recommended learning path for official resources:

  1. Release notes → what's possible now
  2. Claude Help Center → product paths (Cowork, Skills, Connectors)
  3. Claude Code Docs → advanced features (plugins, hooks, agent teams)
  4. API / Agent SDK docs → building custom agent applications
  5. GitHub repositories → real examples and structures
⚠️ Common Mistake: Reading demo posts as specifications, or official docs as operational tips. Keep the roles separate: official docs confirm facts and boundaries; community posts reveal patterns and failure modes; repositories show actual structure.
