Glossary & Quick Reference

Appendix

These terms describe how information is gathered, managed, and degraded within Claude's working memory.

TermDefinition
Context AssemblyThe process of gathering rules, files, memos, and tool descriptions needed for the current task into the actual input context. A selection process, not a dumping process.
Context RotThe phenomenon where early signals weaken as the context grows longer. Long conversations gradually lose coherence because important early instructions get diluted by later content.
Memoized ContextBackground information held across turns so Claude doesn't need re-briefing each time. "Computed once, reused many times."
Prompt CacheA mechanism that provides reuse benefits when similar context prefixes are sent repeatedly. Stable prefixes (CLAUDE.md, rules) should remain consistent to maximize cache hits.
Token EconomyThe discipline of deciding what information stays permanently loaded, what loads on demand, and what gets trimmed — balancing quality against cost.
Tool Result BudgetThe practice of controlling how much context space tool outputs consume. Long logs and verbose results eat the context window and degrade quality.
📌 Key Concept: Context Assembly is not "more is better." It is the selective process of including only what matters for the current task. Tool Result Budget is not optional — it is what prevents a single verbose test run from consuming half your context window.
TermDefinition
Resumable SessionThe ability to pick up where you left off after disconnection or session change, using saved state files.
Resume HygieneThe habit of cleaning up state before resuming — keeping only what the next session needs and discarding accumulated noise.
Session MemoryStructured state preserved for continuation: current status, important files, workflow, errors, and results. Not the raw conversation transcript.
HandoffA formatted memo left for the next session or person, containing current state, next steps, and unresolved risks.
Slash CommandSession management commands like /compact, /clear, /context, /memory, /cost used within Claude Code.
Fork SessionCreating a new session branch when the current direction needs to diverge, preserving the original session state.
Background ExecutionWork that continues after you leave the terminal — the session keeps running without active human monitoring.
TermDefinition
HarnessThe outer structure defining what Claude can read, modify, and verify. Like a colleague's workspace: reference materials, accessible drawers, and a checklist.
Harness EngineeringDesigning the entire work environment — not just better prompts, but the rules, permissions, files, and verification structures Claude operates within.
CLAUDE.mdThe project contract file — always loaded, contains execution commands, forbidden boundaries, and structural descriptions.
WorktreeA separate working directory from the same repository, used for parallel work without file conflicts.
Plan ModeA pattern where exploration and execution are separated — Claude plans before editing, reducing wrong-assumption implementations.
Review SplitSeparating implementation and review into different sessions to reduce context pollution and approval fatigue.
Agentic LoopThe recurring cycle: read → judge → use tools → continue. Work that spans multiple turns, not a single question-answer pair.
TermDefinition
SkillA reusable procedure file (folder) that standardizes repeated tasks. Contains instructions, scripts, references, and assets.
PluginA deployable package bundling skills, commands, hooks, and connection settings. The "onboarding box" for teams.
MCP (Model Context Protocol)The standardized connection protocol for Claude to reach external tools and services. Like USB-C for AI tool integration.
ConnectorA user-facing interface for connecting to external services (Notion, Drive, Slack). Simpler than MCP, less control.
HookAn automatic intervention rule at specific moments (pre-tool-use, post-edit, pre-push). Turns "remember to check" into "always check."
SubagentA secondary worker within the current session with isolated context. Used for research, review, or risk-checking without polluting the main session.
Agent TeamMultiple sessions operating in parallel with message exchange and task allocation — like a real team with a lead and members.
Control SurfaceWhere users can adjust work behavior — not a feature list, but the points where you can intervene in the workflow.
💡 Pro Tip: Quick decision framework: "Same format every time?" → Skill. "Need external data?" → MCP/Connector. "Deploy to the whole team?" → Plugin. "Must verify automatically?" → Hook. "Need parallel context?" → Subagent/Worktree.
TermDefinition
Permission GateThe boundary between allow, ask, and deny — evaluated before any action executes.
Approval GateA checkpoint requiring human sign-off before the next step proceeds. Critical for external communications, deployments, and data modifications.
AllowlistA policy that only passes specifically permitted items (commands, users, domains). Safer than blocking specific threats.
AuditabilityThe ability to explain later: what was read, what was changed, who approved, and what remains. The foundation of sustainable automation.
Alignment FakingA research concept where a model appears to follow instructions while internally maintaining different objectives. The practical takeaway: don't trust model compliance alone — build external verification.
AutonomyHow far work can proceed without human intervention. Higher autonomy requires stronger verification, approval, and permission boundaries.
AgencyHuman initiative in choosing what to delegate, when, and with what constraints. Not about giving more commands — about making better delegation decisions.
Tool OrchestrationThe layer deciding which tools to invoke, in what sequence, and with what parameters.
📌 Key Concept: Observability (seeing system state via dashboards and analytics) and auditability (explaining what happened and why after the fact) are different. You need both: observability for operational efficiency, auditability for accountability and trust.
TermDefinition
LLM OSA metaphor for the emerging work layer: context, tools, permissions, and memory surrounding a model, analogous to how an OS manages programs, files, permissions, and devices.
Software 3.0The concept that natural language itself is becoming a programming interface — work instructions are code.
Vibe CodingGenerating code through natural language, extended now to include testing, reviewing, refactoring, documentation, and deployment.
Remote ControlA human resuming or steering a running session from a distance (different device, different location).
ChannelsExternal systems pushing events into Claude (webhooks, messages) — the trigger comes from outside, not from a human.
Scheduled TasksTime-triggered automation — cron-like execution where the clock starts the work.
Agent SDKDeveloper tools for building custom agent applications with Claude as the execution engine inside your own product.
Adaptive ThinkingClaude adjusting its reasoning depth based on task complexity — thinking longer for hard tasks, shorter for simple ones.
💡 Pro Tip: When confused by a term, ask: "Is this about how work flows (mechanics), how mistakes are prevented (safety), or how sessions persist (continuity)?" This three-bucket mental model covers the vast majority of Claude Code terminology.
