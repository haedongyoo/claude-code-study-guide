When you explain the same procedure for the tenth time — meeting note format, weekly brief structure, code review checklist — it is time to create a Skill. A Skill is a reusable work manual that reduces explanation cost.

📌 Key Concept: Skills are not prompt snippets. They are folder structures with a main document (SKILL.md), supporting scripts, reference files, and assets. The official guide recommends progressive disclosure: the frontmatter and description load first; details load only when the skill activates.
Promote to Skill When...Keep as Conversation When...
Output format repeats, variation needs to shrink, storage location and tool order need fixingThe topic changes often, it's a one-time brainstorming exploration

Projects vs. Skills: Projects are background knowledge that's always present. Skills are procedures activated when a matching task arrives. Background without procedure leads to inconsistent output; procedure without context leads to empty results.

⚠️ Common Mistake: The skill's description field is not an introduction paragraph — it is a routing rule. If the description is vague, the skill either never triggers (undertriggering) or fires for unrelated tasks (overtriggering). Write descriptions like: "Use this skill when the user asks for a weekly competitor brief, market review, or industry trend summary."

Skills cluster into a few common roles regardless of their specific domain:

  • Library/SDK reference — Fixes usage patterns for internal or tricky external libraries
  • Verification/testing — Standardizes checking procedures (Playwright flows, CLI tests)
  • Data pipeline — Organizes data sources and report paths
  • Recurring operations — Weekly reviews, standups, ticket creation
  • Scaffolding — Generates initial service or handler structures
  • Quality gate — Review standards, test rules, security checklists
💡 Pro Tip: Skills that try to do everything ("research, code, test, and deploy") seem powerful but are fragile. Skills that solve one bottleneck precisely survive longest. Verification skills often have higher ROI than generation skills.

Skill Design Template:

---
name: weekly-brief
description: >
  Use for weekly competitor briefs, recurring market checks,
  and industry trend summary requests.
disable-model-invocation: false
---
# Goal
Create a weekly competitor brief using only the last 7 days of evidence.

# Input Sources
- Competitor list: `references/competitors.md`
- Template: `assets/weekly-brief-template.md`

# Steps
1. Read competitor list and template
2. Search for changes in the last 7 days
3. Draft brief following template structure
4. Flag uncertain items separately
5. Save to `outputs/weekly-brief-YYYY-MM-DD.md`

# Rules
- Max 3 action items
- Sources and dates required
- No speculation — mark unknowns as "Needs verification"

A Plugin bundles multiple skills, commands, hooks, and connection settings into a single deployable package. If a Skill is a work manual, a Plugin is the onboarding box you hand to a new team member.

📌 Key Concept: The word "plugin" means different things depending on the surface. Claude Code plugins bundle commands, agents, hooks, skills, MCP, and LSP for development workflows. Cowork plugins bundle skills, connectors, slash commands, and sub-agents for knowledge work roles. Same name, different installation experience.

The progression: MCP connects to external tools → Skills define procedures → Plugins package everything for deployment. In one line: Skills are "how to do it," MCP is "the path to reach it," Plugins are "the box that deploys it."

⚠️ Common Mistake: Treating external Skills and Plugins as just documents. They can contain tool calls, file access, external URL fetches, and execution scripts. Before installing any external skill, check: (1) Does it fetch external URLs? (2) Does it use Bash, file writes, or network? (3) Do its files match its stated purpose? (4) Will it run near sensitive data? (5) Can you disable it quickly?

Model Context Protocol (MCP) is the standardized way Claude connects to external tools and services — like a USB-C port for AI tool integration.

MCP vs. Connector:

ConnectorMCP
User-facing — search and connect from a settings screenDeveloper-facing — design server, tool interface, and response format
Like an app store connectionLike designing the internal wiring panel
Best when connection itself is the goalBest when you need control over tool structure post-connection
💡 Pro Tip: Quick decision: "I just want to connect and use it" → Connector. "I want to control which tools are exposed, response formats, and which projects activate it" → MCP. Mixing them up leads to either over-engineering simple connections or under-engineering complex automation.

Hooks transform "remember to check" into "always check." They are automatic intervention rules at specific moments — pre-tool-use, post-edit, pre-push, etc.

If prompts are road signs ("please drive carefully"), hooks are seatbelt alarms and automatic braking. Signs can be ignored; safety systems change actual behavior. This is why hooks matter more as teams scale.

📌 Key Concept: Hook types include command hooks (shell scripts), prompt-based hooks, agent-based hooks, and HTTP hooks. For teams, managed-only hooks and HTTP hook allowlists provide organizational control that goes far beyond personal shell scripts.

Subagents isolate context. When the main session handles exploration, implementation, review, and documentation, context gets noisy. Delegating research or review to a subagent keeps the main session focused. Think: team lead delegates fact-checking to an analyst instead of doing it personally.

Remote Control / Channels / Scheduled Tasks all re-start work, but the trigger differs:

FeatureWho Pushes the Button
Remote ControlA human, from a distance
ChannelsAn external system (webhook, message)
Scheduled TasksTime itself (cron-like)
💡 Pro Tip: Four practical rules: (1) If you keep repeating the same explanation, extract it to a Skill or rule file. (2) If you keep connecting external tools, narrow permission scope before adding more MCP connections. (3) If verification keeps being forgotten, push it to hooks. (4) If you need parallelism, try a small subagent or worktree before launching full agent teams.
