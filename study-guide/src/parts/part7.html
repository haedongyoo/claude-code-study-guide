Part 7: Governance & Final Practice

Chapters 11–12

Governance is not a compliance checkbox — it is the structure that lets teams use AI with confidence instead of anxiety. Without governance, every automation is a demo that cannot become infrastructure.

📌 Key Concept: The core question governance answers: "What can Claude do without asking, what must it ask about, and what must it never do?" This is the allow/ask/deny framework applied at organizational scale.

Risk classification example:

Risk LevelExamplesApproval Mode
LowMeeting summaries, internal drafts, test additionsAuto-execute
MediumCustomer-facing drafts, regular reports, refactoringHuman review then execute
HighProduction deployment, customer emails, accounting entries, permission changes, sensitive data queriesHuman approval required — no execution without it
💡 Pro Tip: Governance rollout works best in phases: Week 1 — block .env access and destructive commands, require approval for deployments. Week 2 — standardize allowed plugins/skills, organize CLAUDE.md and rules per project. Week 3 — add cost monitoring and establish handoff/approval logging habits.

The permission model uses three modes: allow (proceed automatically), ask (pause for human decision), and deny (block completely). Rules are evaluated in order: deny → ask → allow, with the first match applying.

The four governance layers:

  1. Document layer — CLAUDE.md, rule files, skill instructions: what Claude should know and follow
  2. Settings layer — settings.json, permission rules, managed settings: what Claude can and cannot do
  3. Runtime layer — Hooks, approval gates, plan mode: what happens during execution
  4. Trace layer — Work logs, approval logs, decision logs, handoff files: what happened and why
⚠️ Common Mistake: Using bypassPermissions as a team default. Community posts often recommend "turn off all permissions for speed." Anthropic treats this as a permission mode that organizations can disable entirely via disableBypassPermissionsMode. Individual experiments and team standards should never share the same permission level.

Two-layer permission structure for teams:

  • Global (user-level): Block rm -rf, secret files, force pushes
  • Project-level: Block migration folders, deployment scripts, payment configs specific to this repository

Human-in-the-loop design means building explicit checkpoints where humans make decisions — not hovering over every action. The goal is strategic intervention, not constant supervision.

📌 Key Concept: Auditability means being able to explain later: what was read, what was created, who approved it, and what remains unfinished. Same automation with audit trails becomes infrastructure; without them, it remains a demo.

Minimum audit trail for high-stakes work:

1. source/          — Original materials or links
2. outputs/         — Claude's drafts and revisions
3. work-log.md      — What was executed and when
4. approval-log.md  — Who approved what and when
5. handoff.md       — Remaining risks and next steps

Operations logs vs. decision traces: Operations logs record what happened (commands, tool calls, timestamps). Decision traces record why (what was considered, what alternatives were rejected, why this path was chosen). Agents working on long tasks need both. Without decision traces, you can see where failure occurred but not why.

Incident review template:

# Incident Review
- When was it discovered?
- Which automation / session was involved?
- What were the input materials?
- Which approval point was missing?
- What rule, hook, or permission needs to change?
💡 Pro Tip: Cowork conversations are stored locally and do not appear in Audit Logs, Compliance API, or Data Exports. Do not use Cowork for regulated workloads. For audit-critical work, use Claude Code with Git-based logging, hooks, and external approval tracking.

Team adoption diverges from individual use the moment you start deploying shared rules and assets. It is not about distributing a tool — it is about deploying an operating environment with policies attached.

Key team/enterprise capabilities:

  • Skills provisioning — deploy shared skills organization-wide
  • Managed settings — admin-enforced configurations that individuals cannot override
  • Plugin marketplace management — control which plugins are available
  • Usage analytics dashboard and Analytics API
  • Seat policies — who gets access to which features
📌 Key Concept: Standardization does not kill creativity — it kills repetitive explanation cost. When shared meeting summary skills, client response templates, PR review skills, and weekly brief templates are deployed team-wide, people spend less time on format and more time on judgment.

External skill adoption flow for organizations:

  1. Verify the source
  2. Review included files and tool scope
  3. Test in a sandbox or isolated workspace
  4. Reduce permissions to the minimum needed
  5. Only then deploy to team/organization-managed distribution

Cost as a governance signal: Usage data is not just for finance — it is a design feedback loop. "This agent re-reads a long regulation file every session" → investigate caching. "This automation produces verbose tool output" → trim the output budget. Cost management is operational design, not budget policing.

The final chapter brings everything together with two practical projects: building a small website and building an automation workflow. Both demonstrate that thin, completed results are more valuable than elaborate unfinished plans.

Website project approach:

  1. Fix three lines first: Reader (who reads this?), Goal (what should they understand?), Action (what should they do?)
  2. Create file set: brief.md, tone.md, references/, site/
  3. Use Cowork to draft structure and copy, then Claude Code to implement
  4. Split into three prompts: design (structure), implement (build), review (verify)

Automation project approach:

  1. Define input boundaries and output contract before "when to run it"
  2. Success = same format on re-run, not just first-time success
  3. High-risk steps must route through human approval
📌 Key Concept: Five directions the ecosystem is heading: (1) Conversational → asynchronous operations, (2) Pop-up approval → policy-based approval, (3) Long transcripts → structured task state, (4) Single surface → multi-surface continuity, (5) Prompt tricks → measurable operations with analytics and evaluation loops.

Deployment checklist:

  • Primary work surface chosen (terminal, IDE, desktop, web)?
  • CLAUDE.md and rules separated by role?
  • Permissions and hooks covering real risk boundaries?
  • plan.md, verify.md, handoff.md in place?
  • Review split between implementation and review?
  • Cost and failure measurement established?
  • Scope separation (user/project/local/managed) for teams?
💡 Pro Tip: The book's conclusion is simple: people who use Claude Code well are not people who write long prompts — they are people who stabilize context, manage tokens and cache, narrow permissions, and make review splits and handoffs into habits. One thin completed result plus the structure to recreate it is worth more than any number of impressive demos.
